# 项目复盘文档：下沉市场首套房决策助手

> **复盘框架**: STAR 法则 | **作者**: 产品经理 | **日期**: 2026-07


---

## 一、Situation — 背景与问题

### 市场背景

中国三四线城市（"下沉市场"）的购房者面临严重的决策信息不对称：

| 痛点 | 具体表现 |
|------|----------|
| **平台错配** | 贝壳/链家面向一线城市，"筛选+浏览"模式假设用户已有目标，不解决"不知道买什么合适"的问题 |
| **数据缺失** | 三四线城市房源数据覆盖参差不齐，年份缺失率高达46%，房龄评分直接失效 |
| **金融盲区** | 首次购房者不知道落地成本（首付+税费+中介费+装修），只知道挂牌价，预算评估偏差可达30% |
| **决策困难** | 海量房源中无法判断"哪些真正适合我"，缺乏个性化排序和风险提示 |

### 核心矛盾

> 下沉市场首次购房者的核心矛盾：**在海量房源中，不知道哪些是"自己真正买得起"且"适合自己"的。**

### 为什么是我的项目

- 贝壳/链家做的是"帮你找到房"，我做的差异化是"帮你做决定"
- 从"总价"到"落地价"（含首付+税费+装修），从"筛选"到"推荐"（多维度个性化评分），从"浏览"到"决策"（可导出报告）
- 产品形态不是功能堆砌，而是先写完整PRD（10章：背景→用户画像→竞品分析→功能矩阵→指标体系→路线图→风险），再按优先级开发


## 二、Task — 目标与约束

### 产品目标

**让下沉市场的首次购房者，像有一个私人购房顾问一样，知道自己能买得起什么、该买什么、为什么。**

### 成功标准

| 维度 | MVP 标准 | 衡量方式 |
|------|----------|----------|
| 功能完整性 | 购买力评估 → 房源推荐 → 决策报告 形成闭环 | 5分钟完整演示 |
| 数据准确性 | 首付/月供计算与银行实际误差 <5% | 对照银行计算器验证 |
| 差异化可见 | 与贝壳/安居客有 3+ 个独特功能 | 竞品矩阵对照 |
| 可演示性 | 完整演示"输入→推荐→报告"全流程 | 演示脚本验收 |

### 约束条件

- **技术栈**: Python + Streamlit（轻量、可快速迭代、适合数据产品原型）
- **数据源**: 房天下（fang.com）二手房数据，自建爬虫获取
- **部署**: Streamlit Cloud（免费、自动同步 GitHub）
- **时间**: MVP 4 周内完成闭环
- **资源**: 单人开发（产品+开发+数据一体化）

### 决策一：为什么选择线性加权而非 AHP/TOPSIS

**决策背景**: 评分算法是推荐系统的核心，PRD 路线图中提到 Phase 2 可升级为 AHP + TOPSIS。

**两种方案的权衡**:

| 维度 | 线性加权（当前方案） | AHP + TOPSIS（备选） |
|------|---------------------|---------------------|
| 用户理解成本 | 低 — 拖6个滑块，直观 | 高 — 两两比较矩阵（"价格比面积重要多少？1-9分"） |
| 计算透明度 | 高 — 每维得分×权重=总分 | 低 — 特征值、正负理想解，黑盒 |
| 针对目标用户 | **下沉市场首次购房者，非技术背景** | 学术研究或企业采购场景 |
| 结果差异 | — | 与线性加权 Top5 重合度通常 >85% |

**决策**: 不做。**在推荐场景中，用户信任的不是算法的数学优雅性，而是推荐理由的可理解性。** "价格优(100) + 面积匹配(100) = Top1"比"TOPSIS 相对贴近度 0.87"更有说服力。

**这个决策本身**是产品经理的核心能力判断——知道什么时候用简单方案解决复杂问题。

### 决策二：为什么做数据回填而不是硬藏问题

**决策背景**: 三四线城市年份覆盖率平均 56%，用户画像城市浏阳仅 23.5%。如果直接忽略缺失年份的房源，推荐结果会丢失近一半数据。

**方案对比**:

| 方案 | 优点 | 缺点 |
|------|------|------|
| A: 直接剔除缺年份的房源 | 简单 | 丢弃 49.7% 数据，推荐覆盖面大幅缩水 |
| B: 房龄维度不参与评分 | 不丢数据 | 牺牲评分准确性，用户感知不到 |
| **C: 三层方案（采用）** | 数据+算法+UI 协同 | 实现复杂度高 |

**方案 C 的三层设计**:

```
第一层（核心防御）：算法自适应降级
  → 覆盖率 < 70%：房龄权重自动减半
  → 覆盖率 < 50%：房龄权重降为 0
  → 差额自动分配至其他 5 个维度

第二层（根源修复）：渐进式数据回填
  → 用户查询城市优先回填（埋点记录 → 驱动优先级）
  → 断点续传，支持分批执行

第三层（透明化）：UI 标注数据质量
  → A/B/C/D 四级质量标签
  → 自动降级时告知用户原因
```

**决策理由**: 第一层防数据问题影响推荐质量，第二层根治数据缺失，第三层建立用户信任。**三层不是替代关系，是分层防御**——回填解决根源问题需要时间（几十小时），算法降级在回填完成前兜底，UI 透明化确保用户知情。

### 决策三：为什么"月供占可支配收入比"比"月供占收入比"更准

**决策背景**: 购买力评估中，月供安全线通常定义为"月收入 × 50%"。但下沉市场用户月支出现金流不可忽略——一个家庭月入10000但每月支出5000，实际能用于月供的只有5000。

**改进**: 表单增加月支出字段（选填），填写后使用"可支配收入 = 月收入 - 月支出"作为月供计算基准。不填则使用传统方法。

**实测效果**: 月入10000/月支5000 → 舒适线从79万降到55.3万，**避免推荐超出实际承受能力的房源**。

### 决策四：为什么不做持有成本模拟

**决策背景**: PRD 路线图中 Phase 2 规划了"10年持有成本模拟（利息+物业+装修折旧）+ 租房 vs 买房对比"。

**决策**: 不做。理由有三：

1. **下沉市场买房是刚需**——不存在"我算算划不划算然后决定不买了"。计算器回答的问题（"10年利息多少"）不是需要决策辅助的问题（"买哪套"）。

2. **不改变推荐结果**——不管利息高低，推荐的还是同一批房源。一个不改变输出的功能，对用户没有增量价值。

3. **面试减分风险**——面试官看到持仓模拟会问"你的用户真的需要这个吗？"——答不上来比没有这个功能更糟糕。

**这个"不做"的决策，和做 AHP、A/B 测试、反馈投票是一样的逻辑——在资源有限的 MVP 阶段，只做影响决策结果的功能。**

## 四、Result — 成果与反思

### 可演示的闭环

```
用户打开平台
  → 首页加载46,625条房源数据（511个城市）
  → 进入"决策报告"
  → 填写购房偏好（预算/面积/户型/城市）+ 收入/储蓄/月支出/首套二套
  → 调整6维权重滑块
  → 点击"生成报告"
  → 看到：
     · 买得起总价（舒适线 + 安全线）
     · 首付现金需求
     · Top20推荐列表（含月供、月供占比、适配等级）
     · Top3评分归因（这套为什么排第一）
     · 各维度得分分解图
     · 投资评估（城市趋势 + 单价分析 + 流动性）
     · 5类风险检测结果
  → 下载PDF决策报告
```

### 如果我有 DAU...

如果产品有真实用户，我会关注以下指标来驱动迭代：

| 指标 | 为什么重要 | 优化方向 |
|------|-----------|----------|
| 购买力评估完成率 | 用户是否愿意填收入信息？ | 默认值优化（基于城市中位数预填） |
| 报告生成率 | 从填表到生成报告的转化 | 表单过长可能是障碍 |
| Top3推荐CTR | 用户是否点击推荐房源查看详情？ | 评分归因文字是否有效 |
| PDF下载率 | 用户是否觉得报告值得保存？ | 报告格式和信息密度 |
| 次日留存率 | 用户会回来重新跑分析吗？ | 推荐质量是否让用户信任 |

### 技术反思

**做得好的**:
- Streamlit 选型正确 — 7个 Python 文件完成完整闭环，适合数据产品快速验证
- 自适应权重降级 — 用工程手段解决了数据质量问题，而非假装数据没问题
- 三层防御体系（算法→回填→UI 透明化）— 每层独立运作，互不阻塞

**可以改进的**:
- 数据时效性 — 当前依赖手动运行爬虫，缺乏自动化更新。如果数据过期超过3个月，推荐的可信度会下降
- 缺乏配套数据 — POI（学校/医院/商场）、通勤时间、城市发展规划等维度能显著提升推荐质量，但需要接入外部 API
- 移动端体验 — Streamlit 对移动端支持有限，下沉市场用户更多用手机

### 产品反思

**最重要的一条经验**: **做减法比做加法更需要判断力。**

这个项目从 PRD 的 12 个功能项，实际只做了 7 个——砍掉了 AHP、A/B 测试、持有成本模拟、反馈投票。每一次砍掉的理由都不同，但核心逻辑一致：**只做影响决策结果的功能，不做可有可无的"加分项"。**

对面试来说，能解释"为什么不做"比"做了什么"更能体现产品判断力。
