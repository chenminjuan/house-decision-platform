## 2. 指标详细定义

### 2.1 北极星指标

> **"完成一次完整的决策报告生成并下载PDF的用户数"（周维度）**

**为什么是这个指标？**

- 代表用户完成了从"输入信息"到"拿到可执行建议"的完整闭环
- PDF 下载意味着用户认为这份报告"值得保存/分享/打印"——是真实价值的信号
- 单一指标，团队不会在多个指标之间产生分歧

**备选北极星淘汰理由**：

| 候选指标 | 淘汰理由 |
|----------|---------|
| DAU | 太浅，来了不用的用户没有价值 |
| 房源浏览量 | 浏览多是信息消费，不代表决策价值 |
| 停留时长 | 太长可能意味着"找不到想要的东西" |
| 搜索次数 | 搜索多 = 推荐不够好 |

**北极星的健康度量**：
- 如果北极星在涨但 NPS 在跌 → 用户被迫用（渠道驱动）而非真心觉得好
- 如果北极星在涨但留存率在跌 → 一次性工具，没有复购/复用价值

---

### 2.2 获取指标（Acquisition）

| 指标 | 定义 | 统计周期 | 目标（MVP） | 数据来源 |
|------|------|:---:|------|------|
| **DAU** | 日活跃用户（至少访问1个页面） | 日 | >100 | Streamlit 统计 |
| **WAU** | 周活跃用户 | 周 | >500 | Streamlit 统计 |
| **访问来源** | 直接/搜索/社媒/推荐 | 周 | - | 自行埋点 |
| **新用户占比** | 首次访问 / 总DAU | 日 | >60% | Session State |

**MVP 阶段的务实做法**：
- Streamlit 本身不提供用户级分析，MVP 阶段可以用 `session_state` 记录关键事件到本地 log 文件
- Phase 3 考虑接入 Google Analytics 或自建埋点

---

### 2.3 激活指标（Activation）

| 指标 | 定义 | 目标（MVP） | 重要性 |
|------|------|:---:|:---:|
| **购买力评估完成率** | 完成"输入收入/储蓄→看到计算结果"的用户 / 访问用户 | >40% | ★★★★★ |
| **偏好设置完成率** | 完整填写购房偏好表单 / 开始填写 | >70% | ★★★★☆ |
| **首次推荐查看率** | 看到推荐列表 / 完成偏好设置 | >80% | ★★★☆☆ |
| **激活时间** | 从首次访问到完成购买力评估的时间 | <3分钟 | ★★★☆☆ |

**激活的定义（"Aha Moment"）**：
> 用户完成购买力评估并看到第一条个性化推荐结果。

这个时刻用户会意识到："这个东西和贝壳不一样，它知道我能买得起什么。"

---

### 2.4 留存指标（Retention）

| 指标 | 定义 | 目标（MVP） | 基准参考 |
|------|------|:---:|------|
| **次日留存率** | D0 生成报告的用户中，D1 回访的比例 | >30% | 工具类产品中位~25% |
| **7日留存率** | D0 生成报告的用户中，D7 回访的比例 | >15% | 工具类产品中位~12% |
| **周活跃天数** | 用户每周平均使用天数 | >1.5天 | - |

**留存分析框架**：

```
用户为什么回来？
├── 短期（1-3天）
│   ├── 回去和伴侣/家人讨论报告结果 → 回来调整参数再看
│   └── 看完第一份报告后想对比另一个城市
│
├── 中期（1-4周）
│   ├── 实际去看了推荐的房子 → 回来反馈/重新评估
│   ├── 新上了一批房源 → 提醒用户回来看
│   └── 利率/政策变化 → 重新计算月供
│
└── 长期（1月+）
    ├── 完成购房 → 可能会推荐给朋友
    └── 换房需求 → 几年后再次使用
```

**圈定策略**：
- "您上次看中的XX小区新上架了3套房源，要看看吗？"（有数据更新时触发）
- "距离您上次生成报告已过去2周，房价可能有变化，要重新评估吗？"

### 2.5 转化指标（Conversion/Revenue）

| 指标 | 定义 | 目标（MVP） | 说明 |
|------|------|:---:|------|
| **报告生成率** | 生成决策报告的用户 / 完成偏好设置的用户 | >50% | 漏斗最大流失点，需重点优化 |
| **PDF下载率** | 下载PDF / 生成报告 | >30% | 核心价值信号 |
| **推荐有用率** | 标记"有用" / 总反馈 | >60% | 推荐质量的直接反馈 |

**核心转化漏斗**：

```
访问首页            100%
   ↓
加载数据             ~80%    （20%流失：加载太慢/不知道点哪里）
   ↓
完成购买力评估       ~50%    （30%流失：不想输入收入信息/价值感不够）
   ↓
填写购房偏好         ~40%    （10%流失：表单太长/不知道权重怎么调）
   ↓
生成决策报告         ~25%    （15%流失：等待计算/结果不如预期）
   ↓
下载PDF              ~10%    （15%流失：不需要保存/不知道下载按钮在哪）
```

**目标**：将"访问→下载PDF"的漏斗转化率从预估10%提升到15-20%。

### 2.6 推荐指标（Referral）

| 指标 | 定义 | 目标 | 说明 |
|------|------|:---:|------|
| **NPS** | "你有多大可能向朋友推荐这个工具？"（0-10分） | >30 | 产品核心健康指标 |
| **社媒提及** | 小红书/朋友圈/微信群中被提及的次数 | 定性追踪 | 间接衡量口碑 |
| **直接分享率** | 点击"分享"按钮 / 下载PDF | >15% | Phase 2 加分享功能 |

## 3. 数据质量指标

这些指标不直接面向用户，但直接影响用户体验——如果数据质量差，推荐结果就不会好。

| 指标 | 当前值 | 目标值 | 优先级 | 说明 |
|------|:---:|:---:|:---:|------|
| 建成年份覆盖率 | 54.4% | >70% | P0 | 缺失率过高影响房龄评分 |
| 房源去重率 | ~99% | >99.5% | P1 | origin_url 去重逻辑验证 |
| 异常值比例 | <5% | <3% | P1 | 价格/面积/单价异常值过滤 |
| 城市最低挂牌量 | 部分<10 | 分析时>30 | P0 | 挂牌量太少的城市统计不可靠 |
| 数据时效性标注 | 无 | 有 | P1 | 让用户知道数据是什么时候的 |
| 数值一致性 | 未知 | 单价≈总价×10000/面积 偏差<5% | P1 | 交叉验证原始数据质量 |

## 5. 指标拆解到功能

每个功能上线前，需要明确"这个功能影响哪个指标？预期提升多少？"

| 功能 | 影响指标 | 预期提升 | 验证方式 |
|------|---------|:---:|------|
| 购买力评估引擎 | 激活率、报告生成率 | 激活率+15% | A/B 有/无购买力评估 |
| 落地成本计算器 | PDF下载率 | 下载率+10% | 报告中是否包含落地成本 |
| 敏感度分析 | 留存率 | 次日留存+8% | 观察有/无使用敏感度分析的用户留存差异 |
| 对比功能 | 停留时长、留存率 | 7日留存+5% | 使用对比功能的用户留存 |
| 推荐反馈按钮 | 推荐有用率 | - | 直接观察 |
| 分享按钮 | 新用户占比 | 新用户+5% | Phase 2 功能 |

## 6. MVP 阶段的务实做法

MVP 不建议投入过多精力在埋点基建上。采用"够用就好"的原则：

1. **Streamlit session_state**：记录关键用户行为到本地日志
2. **每周手动 review**：看日志、看用户反馈
3. **定性优先于定量**：MVP 阶段 100 个用户中，10 个深度用户的反馈比统计数据更有价值
4. **Phase 3 再建正式的数据看板**

---

> "你无法改进你无法衡量的东西。" —— 但 MVP 阶段，有方向性的定性判断比精准的错误指标更重要。
