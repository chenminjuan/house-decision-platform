# 产品需求文档：下沉市场首套房决策助手

> **版本**: v1.0 | **作者**: 产品经理 | **日期**: 2025-07 | **状态**: 规划阶段

---

## 1. 背景与问题定义

### 1.1 市场背景

中国三四线城市（"下沉市场"）的房地产市场与一线城市存在本质差异：

- **信息不对称更严重**：贝壳、链家等平台在一线城市覆盖充分，但在三四线城市数据质量参差不齐，缺乏专业的购房决策工具
- **购买力约束更强**：三四线城市居民月收入普遍在 3,000-8,000 元区间，房价 30-150 万为主流，月供占收入比是核心决策变量
- **产品形态错配**：现有房产平台的"筛选+浏览"模式面向的是"已有目标的搜索行为"，而非"不知道买什么合适"的决策辅助场景
- **金融知识缺口**：首付计算（契税/个税/增值税/中介费）、月供计算（等额本息/等额本金）、银行贷款审批红线（月供≤月收入50%）——这些对首次购房者是刚需但缺乏系统引导

### 1.2 核心问题

> **下沉市场首次购房者面临的核心矛盾**：在海量房源中，不知道哪些是"自己真正买得起"且"适合自己"的。

现有平台的痛点：
1. 只看总价，不看"落地成本"（首付+税费+中介费+装修预算）
2. 只看房源，不看"月供压力"（月收入 vs 月供的真实对比）
3. 只有筛选，没有"决策"（排序逻辑单一，缺乏个性化权重）

### 1.3 产品愿景

**让下沉市场的首次购房者，像有一个私人购房顾问一样，知道自己能买得起什么、该买什么、为什么。**

## 2. 目标用户

### 2.1 核心用户画像

| 维度 | 用户A：小王 | 用户B：小张夫妇 | 用户C：老刘 |
|------|-----------|---------------|-----------|
| **年龄** | 26-30岁 | 28-35岁 | 35-45岁 |
| **城市** | 三四线城市（如浏阳、湛江、北海） | 三四线城市 | 县城 |
| **月收入** | 5,000-8,000元 | 家庭8,000-15,000元 | 6,000-10,000元 |
| **储蓄** | 10-20万 | 20-40万 | 15-30万 |
| **购房阶段** | 首次看房，完全小白 | 已看过多套，纠结中 | 改善型但预算有限 |
| **核心焦虑** | "我能买得起多少钱的房子？" | "A小区vs B小区到底选哪个？" | "卖了旧的换新的划不划算？" |
| **决策权重** | 月供>总价>通勤 | 面积>学区>月供>通勤 | 面积>总价>持有成本 |

> 详见 [user_personas.md](./user_personas.md)

### 2.2 非目标用户

- 投资型购房者（多套房持有者）：这不是一个投资回报计算器
- 一线城市购房者：数据覆盖不足，贷款逻辑不同（额度、政策差异大）
- 已有明确目标的搜索型用户：贝壳/链家更适合

| **购买力评估** | 无 | 无 | 无 | 无 | **核心功能** |

### 3.2 差异化定位

**贝壳/链家做的是"帮你找到房"，我们做的是"帮你做决定"。**

关键差异：
1. **从"总价"到"落地价"**：首付+税费+中介费+装修预算 = 真实上车门槛
2. **从"筛选"到"推荐"**：基于用户收入和偏好的个性化评分，而非普适筛选
3. **从"浏览"到"决策"**：输出的是可执行的决策报告，而非房源列表

## 4. 核心功能

### 4.1 功能全景

```
首套房决策助手
├── P0 - 购买力评估引擎
│   ├── 落地总成本计算器（首付+税费+中介费）
│   ├── 月供计算器（等额本息/等额本金）
│   └── 购买力边界分析（"你的预算真正能买到什么"）
│
├── P0 - 个性化房源匹配
│   ├── 硬筛选（预算/面积/户型/城市/朝向/房龄）
│   ├── 多维度加权评分（价格/面积/房龄/单价/地段）
│   └── 用户自定义权重
│
├── P1 - 决策报告生成
│   ├── Top N 推荐列表（含得分分解）
│   ├── 场景化风险提示（老旧/高价/低流动）
│   ├── 城市趋势分析
│   └── PDF 导出
│
├── P1 - 对比分析
│   ├── 多城市/小区对比
│   ├── 雷达图（多维度可视化）
│   └── "如果...会怎样"敏感度分析
│
├── P2 - 持有成本分析
│   ├── 10年持有成本模拟（利息+物业+装修折旧）
│   ├── 租房 vs 买房长期对比
│   └── 利率变动影响模拟
│
└── P2 - 社区化
    ├── 用户反馈（推荐有用/无用）
    ├── NPS 评分
    └── 数据贡献激励
```

### 4.2 功能详细说明

#### P0-1: 购买力评估引擎

**用户故事**：作为首次购房者，我想知道以我目前的收入和储蓄，真正能买得起多少钱的房子，而不是只看挂牌价。

**核心逻辑**：
```
落地总成本 = 首付 + 契税 + 个税 + 增值税 + 中介费 + 装修预留
首付 = 总价 × 首付比例（首套房通常20-30%）
契税 = 总价 × 税率（90㎡以下1%，90㎡以上1.5%）
个税 = 总价 × 1%（满五唯一免征）
增值税 = 总价 × 5.6%（满二免征）
中介费 = 总价 × 1-2%
装修预留 = 面积 × 装修单价（用户输入，默认800元/㎡）

月供 = (贷款金额 × 月利率 × (1+月利率)^期数) / ((1+月利率)^期数 - 1)
月供安全线 = 家庭月收入 × 50%（银行审批红线）
月供舒适线 = 家庭月收入 × 30%（推荐线）
```

**输出**：
- "你买得起的真实价格上限是 X 万"（而非"预算X万"的粗略概念）
- "推荐月供范围：X元 - Y元"
- "需要准备的首付现金：X万元"

#### P0-2: 个性化房源匹配

**用户故事**：作为购房者，我不想看上千套房源，我只想看最适合我的20套。

**核心逻辑**：
```
Phase 1 - 硬筛选：
  预算范围、面积范围、户型、朝向、城市、房龄、单价上限

Phase 2 - 多维度加权评分（0-100分）：
  - 价格合理性（越低越好，相对预算范围归一化）
  - 面积匹配度（高斯距离到理想面积）
  - 房龄评分（越新越高，40年为0分）
  - 单价合理性（越低越好，相对筛选集归一化）
  - 地段热度（挂牌密度百分位作为流通性代理）

Phase 3 - 排序输出
  按综合得分降序，Top 20 推荐
```

#### P1-1: 决策报告

**用户故事**：我不想自己分析，我想要一份完整的决策报告。

**报告结构**：
1. 购房需求概览（预算/面积/户型/朝向/城市）
2. 购买力评估结果（落地成本+月供分析）
3. 匹配房源概览（数量/均价/分布）
4. Top 20 推荐列表（含得分分解）
5. 各维度得分分解（Top5平均）
6. 投资评估（城市趋势+单价分析+流动性）
7. 风险评估（高价位/老旧/低流动性/数据缺失）
8. 行动建议

#### P1-2: 对比分析

**用户故事**：我在A小区和B小区之间纠结，想知道客观数据怎么说。

**对比维度**：
- 价格对比（均价/中位价/单价）
- 价值对比（面积单价/房龄/朝向）
- 市场对比（挂牌量/流动性/趋势）
- 雷达图可视化
- 敏感度分析："如果你把价格的权重从50%调到80%，Top10会发生什么变化？"

## 5. 指标体系

### 5.1 北极星指标

**"完成一次决策报告生成并下载PDF的用户数"**

这个指标代表用户从"浏览"走到了"行动"，是产品价值的核心体现。

### 5.2 关键指标

| 层级 | 指标 | 定义 | 目标值 |
|------|------|------|--------|
| **获取** | DAU | 日活跃用户数 | 上线首月100+ |
| **激活** | 购买力评估完成率 | 填写收入/储蓄并完成计算 / 访问用户 | >40% |
| **留存** | 次日留存率 | 次日回访 / 首日访问 | >30% |
| **转化** | 报告生成率 | 生成决策报告 / 完成偏好设置 | >50% |
| **转化** | PDF下载率 | 下载PDF / 生成报告 | >30% |
| **推荐** | NPS | 净推荐值 | >30 |
| **质量** | 推荐有用率 | 标记"有用" / 总反馈 | >60% |

### 5.3 数据质量指标

| 指标 | 当前值 | 目标值 | 说明 |
|------|--------|--------|------|
| 建成年份覆盖率 | 54.4% | >70% | 通过交叉验证补全 |
| 城市最低挂牌量 | 200 | >50 | 确保统计显著性 |
| 数据更新频率 | 静态CSV | 月度更新 | 建立数据管道 |

> 详见 [metrics_framework.md](./metrics_framework.md)

## 6. 非功能需求

### 6.1 性能

- 数据加载：10万+数据加载+清洗 < 30秒
- 推荐计算：硬筛选+加权评分 < 3秒
- PDF 生成：< 5秒
- 页面交互：筛选器联动响应 < 1秒

### 6.2 可访问性

- 支持主流浏览器（Chrome / Edge / Safari）
- 部署到 Streamlit Cloud，生成公开可访问 URL
- 移动端响应式（至少基本可用）

### 6.3 数据安全

- 用户个人数据（收入/储蓄）不持久化存储
- 房源数据仅用于展示和分析
- PDF 报告中不含用户敏感信息

## 8. 成功标准

### MVP 阶段的成功定义：

1. **功能完整性**：购买力评估 + 房源推荐 + 决策报告 形成闭环
2. **数据准确性**：首付/月供计算结果与银行实际误差 <5%
3. **差异化可见**：与贝壳/安居客的功能对比中有 3+ 个独特功能
4. **可演示性**：能用 5 分钟向面试官完整演示从"输入信息"到"拿到决策报告"的全流程

### 长期成功定义：

1. 用户从访问到生成报告的转化率 >50%
2. PDF 下载后的次日留存 >30%
3. NPS >30

## 9. 风险与假设

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 数据时效性不足（静态CSV） | 价格/房源可能已过期 | 标注数据更新日期，透明化数据来源 |
| 三四线城市数据量不均 | 部分城市样本太少，统计不可靠 | 挂牌量<30的城市标注"数据有限" |
| 建成年份缺失率高（46%） | 房龄评分失真 | 缺失值单独处理，不参与该维度评分 |
| 缺乏配套数据（POI/交通/学校） | 通勤/学区分析无法实现 | MVP阶段不承诺此功能，phase 3 可接入高德API |
| 用户信任度 | 推荐结果影响重大决策 | 明确标注"仅供参考"，提供风险提示 |
